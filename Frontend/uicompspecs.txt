### 4. COMPONENT IMPLEMENTATION

#### 4.1 KanbanBoardComponent

**Component Type:** Feature

**Mapped From A1:** KanbanBoardComponent (main container for three-column Kanban layout)

**Purpose:** Main container component that manages the three-column Kanban board layout with drag-and-drop functionality

**Parent:** KanbanPageComponent

**Children:**
- KanbanColumnComponent (3 instances)
- TaskModalComponent

**Dependencies:**
- Angular CDK Drag Drop
- KanbanService
- TaskModalComponent

**Library Components Used:**
- Dialog Component → TaskModalComponent wrapper

**TypeScript Specification**

**Inputs:**
- columns: KanbanColumn[] (array of column configurations)
- tasks: Task[] (array of all tasks)
- loading: boolean (loading state indicator)
- error: string | null (error message)

**Outputs:**
- taskMoved: EventEmitter<TaskMoveEvent> (when task is moved between columns)
- taskCreated: EventEmitter<Task> (when new task is created)
- taskUpdated: EventEmitter<Task> (when task is updated)
- taskDeleted: EventEmitter<string> (when task is deleted, emits task ID)

**State:**
- draggedTask: Task | null (currently dragged task)
- columnData: KanbanColumn[] (processed column data with tasks)
- isLoading: boolean (component loading state)
- errorMessage: string | null (error display message)
- isTaskModalOpen: boolean (modal visibility state)
- selectedTask: Task | null (task being edited)

**Methods:**
- ngOnInit(): void → Initialize component and load data
- loadKanbanData(): void → Fetch initial kanban data
- onTaskDrop(event: CdkDragDrop<Task[]>): void → Handle task drop events
- openTaskModal(task?: Task): void → Open task creation/editing modal
- closeTaskModal(): void → Close task modal
- handleTaskSave(task: Task): void → Process task save operations
- handleError(error: any): void → Process and display errors

**HTML STRUCTURE (PSEUDO CODE ONLY)**

```html
<div class="kanban-board-container" 
     role="region" 
     aria-label="Kanban Board" 
     [attr.aria-busy]="isLoading">
  
  <div class="kanban-error" *ngIf="errorMessage" role="alert">
    <span>{{ errorMessage }}</span>
    <button (click)="loadKanbanData()">Retry</button>
  </div>
  
  <div class="kanban-grid" 
       cdkDropListGroup 
       *ngIf="!errorMessage">
    
    <app-kanban-column 
      *ngFor="let column of columnData; trackBy: trackByColumnId"
      [column]="column"
      [tasks]="getTasksForColumn(column.id)"
      [loading]="isLoading"
      (taskDropped)="onTaskDrop($event)"
      (addTaskClicked)="openTaskModal()"
      (taskClicked)="openTaskModal($event)">
    </app-kanban-column>
    
  </div>
  
  <app-task-modal
    [isOpen]="isTaskModalOpen"
    [task]="selectedTask"
    [mode]="selectedTask ? 'edit' : 'create'"
    (taskSaved)="handleTaskSave($event)"
    (modalClosed)="closeTaskModal()">
  </app-task-modal>
  
</div>
```

**CSS SPECIFICATION**

**Layout:**
- .kanban-board-container: flex column, full height
- .kanban-grid: CSS Grid with grid-template-columns: repeat(3, 1fr)
- gap: 1.5rem between columns

**Spacing:**
- padding: 1.5rem on container
- gap: 1.5rem between grid items

**Component Styles:**
- .kanban-error: error state styling with retry button
- .kanban-grid: responsive grid layout

**Responsive:**
- Desktop (>1024px): 3 columns, full layout
- Tablet (768px-1024px): 3 columns, reduced padding
- Mobile (<768px): single column stack, horizontal scroll

**API INTEGRATION**

- loadKanbanData() → GET /youth-accounts/{id}/dashboard → Load initial dashboard data and transform to kanban structure
- handleTaskSave() → POST /tasks (when implemented) → Create new task
- handleTaskSave() → PUT /tasks/{id} (when implemented) → Update existing task
- onTaskDrop() → PUT /tasks/{id}/status (when implemented) → Update task status/column

**ERROR HANDLING**

- API failure → show error message with retry button
- Drag drop failure → revert task position and show snackbar
- Task save failure → show validation errors in modal
- Network timeout → show connection error message

**INTERACTION FLOW**

**User Action → UI → API**
- Load page → ngOnInit() → GET /youth-accounts/{id}/dashboard → display columns and tasks
- Drag task → onTaskDrop() → PUT /tasks/{id}/status → update UI optimistically
- Click "Add Task" → openTaskModal() → show modal → save → POST /tasks → refresh data
- Click task card → openTaskModal(task) → show edit modal → save → PUT /tasks/{id} → refresh data

#### 4.2 KanbanColumnComponent

**Component Type:** Feature

**Mapped From A1:** KanbanColumnComponent (individual column container)

**Purpose:** Represents a single column in the Kanban board (To Do, In Progress, Done) with drag-drop functionality

**Parent:** KanbanBoardComponent

**Children:**
- TaskCardComponent (multiple instances)
- ButtonComponent (Add Task button)

**Dependencies:**
- Angular CDK Drag Drop
- TaskCardComponent
- ButtonComponent (from library)

**Library Components Used:**
- Button Component → Add Task button

**TypeScript Specification**

**Inputs:**
- column: KanbanColumn (column configuration with id, title, color)
- tasks: Task[] (tasks assigned to this column)
- loading: boolean (loading state)
- allowDrop: boolean (whether column accepts drops)

**Outputs:**
- taskDropped: EventEmitter<CdkDragDrop<Task[]>> (when task is dropped in column)
- addTaskClicked: EventEmitter<string> (when add task button clicked, emits column ID)
- taskClicked: EventEmitter<Task> (when task card is clicked)
- menuToggled: EventEmitter<string> (when column menu is toggled)

**State:**
- isDragOver: boolean (drag over state for visual feedback)
- isLoading: boolean (column-specific loading)
- taskCount: number (computed task count)
- isMenuOpen: boolean (column menu visibility)

**Methods:**
- ngOnInit(): void → Initialize column
- onTaskDrop(event: CdkDragDrop<Task[]>): void → Handle task drops
- onAddTaskClick(): void → Emit add task event
- onTaskClick(task: Task): void → Emit task click event
- toggleMenu(): void → Toggle column menu
- getTaskCount(): number → Calculate task count

**HTML STRUCTURE (PSEUDO CODE ONLY)**

```html
<div class="kanban-column" 
     [class.drag-over]="isDragOver"
     [attr.aria-labelledby]="column.id + '-header'">
  
  <div class="column-header" [id]="column.id + '-header'">
    <h3 class="column-title">{{ column.title }}</h3>
    <span class="task-count" aria-label="Task count">{{ taskCount }}</span>
    <button class="column-menu-btn" 
            (click)="toggleMenu()"
            [attr.aria-expanded]="isMenuOpen">
      <i class="icon-more" aria-hidden="true"></i>
    </button>
  </div>
  
  <div class="column-content"
       cdkDropList
       [cdkDropListData]="tasks"
       (cdkDropListDropped)="onTaskDrop($event)"
       (cdkDropListEntered)="isDragOver = true"
       (cdkDropListExited)="isDragOver = false">
    
    <app-task-card
      *ngFor="let task of tasks; trackBy: trackByTaskId"
      [task]="task"
      [draggable]="!loading"
      cdkDrag
      (taskClicked)="onTaskClick(task)">
    </app-task-card>
    
    <div class="empty-state" *ngIf="tasks.length === 0 && !loading">
      <p>No tasks in {{ column.title }}</p>
    </div>
    
    <div class="loading-state" *ngIf="loading">
      <div class="spinner" aria-label="Loading tasks"></div>
    </div>
    
  </div>
  
  <div class="column-footer">
    <app-button
      variant="ghost"
      size="small"
      iconName="plus"
      label="Add Task"
      (userClick)="onAddTaskClick()"
      [disabled]="loading">
    </app-button>
  </div>
  
</div>
```

**CSS SPECIFICATION**

**Layout:**
- .kanban-column: flex column, min-height: 500px
- .column-content: flex-grow: 1, overflow-y: auto
- .column-header: flex row, justify-content: space-between

**Spacing:**
- padding: 1rem on column
- gap: 0.75rem between tasks
- margin-bottom: 0.5rem on header

**Component Styles:**
- .drag-over: border highlight and background change
- .empty-state: centered text with muted color
- .task-count: badge styling with background

**Responsive:**
- Desktop: fixed width columns
- Tablet: flexible width with min-width
- Mobile: full width in stacked layout

**API INTEGRATION**

- onTaskDrop() → Emit to parent → PUT /tasks/{id}/status → Update task column
- No direct API calls (receives data from parent)

**ERROR HANDLING**

- Drop failure → show error toast and revert position
- Loading error → show error state in column
- Empty state → show helpful message and add task prompt

**INTERACTION FLOW**

**User Action → UI → API**
- Drop task → onTaskDrop() → emit to parent → API call → UI update
- Click "Add Task" → onAddTaskClick() → emit to parent → open modal
- Click task → onTaskClick() → emit to parent → open edit modal

#### 4.3 TaskCardComponent

**Component Type:** Feature

<div class="task-card"

      <i class="icon-more" aria-hidden="true"></i>
    </button>
  </div>
  
  <div class="task-content">
    <h4 class="task-title">{{ task.title }}</h4>
    <p class="task-description" *ngIf="task.description">
      {{ task.description | truncate:100 }}
    </p>

**Mapped From A1:** TaskModalComponent (task creation and editing form)

**Purpose:** Modal dialog for creating new tasks and editing existing tasks with form validation

**Children:**

### 8. NOTES & CONSTRAINTS

**HTML Structure Compliance:**
- HTML must follow A1 structure strictly as the single source of truth
- Three-column Kanban layout with CSS Grid implementation required
- ARIA attributes mandatory for accessibility compliance
- Responsive breakpoints: desktop (>1024px), tablet (768px-1024px), mobile (<768px)

**Component Library Usage:**
- Do NOT create new components unnecessarily
- Prefer reusable library components from uicomplibraryplay
- Follow exact import patterns defined in library documentation
- Use Button, Dialog, Chip-Tag, FileUpload, and Breadcrumb components where applicable

**API Integration:**
- Current API (GET /youth-accounts/{id}/dashboard) provides limited data
- Task CRUD operations require future API implementation
- OAuth2 + JWT Bearer Token authentication required
- Error handling must follow API response standardization
- Rate limiting headers should be used for UI feedback

**Development Standards:**
- HTML should be written as structured pseudo-code (not full framework code)
- CSS should include only key classes and layout rules, not full stylesheets
- TypeScript interfaces required for all data models
- Angular reactive forms for all form implementations
- OnPush change detection strategy recommended for performance

**Accessibility Requirements:**
- ARIA labels and roles required for all interactive elements
- Keyboard navigation support mandatory
- Screen reader compatibility required
- Focus management for modals and dropdowns
- Color contrast compliance (WCAG 2.1 AA)

**Performance Considerations:**
- Implement virtual scrolling for large task lists
- Use trackBy functions for *ngFor loops
- Lazy loading for feature modules
- Memoization for expensive calculations
- Image optimization for avatars and icons

**Error Handling Strategy:**
- API failures → show user-friendly error messages with retry options
- Form validation → inline error messages with field highlighting
- Network timeouts → show connection error with offline capability
- Drag-drop failures → revert operations and show feedback
- Missing data → graceful degradation with placeholders
