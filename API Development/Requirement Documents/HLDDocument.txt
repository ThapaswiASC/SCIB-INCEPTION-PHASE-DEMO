## 1. Introduction

### 1.1 Purpose
This High-Level Design document outlines the architecture for implementing real-time task creation synchronization across all user sessions in the SCIB Task Management System. The solution ensures that newly created tasks appear immediately on all connected users' boards within 500ms, maintaining data consistency and providing optimal user experience.

### 1.2 Scope
This document covers:
- Task creation form design and validation
- Backend task creation and broadcast services
- Real-time synchronization mechanisms
- Database constraints and indexing
- Bulk task creation capabilities
- API contracts and error handling

### 1.3 Stakeholders
- SCIB Team Members (Primary Users)
- Development Team
- QA Team
- System Administrators
- Enterprise Architects

---

## 2. System Overview

### 2.1 Business Requirements
**Primary User Story**: As a SCIB team member, I want newly created tasks to appear immediately on all users' boards so that everyone has the most current view of work items.

**Key Requirements**:
- REQ-RT-007: Real-time task synchronization
- REQ-RT-008: Task validation and error handling

### 2.2 Success Criteria
- Tasks appear on all connected users' boards within 500ms
- Complete metadata synchronization (title, description, assignee, priority, due date, tags)
- Specific error messages for validation failures
- Bulk task creation with correct ordering
- No broadcast on validation failures

## 4. Component Design

### 4.1 Frontend Components

#### 4.1.1 TaskCreationFormComponent
**Location**: `src/app/components/task-creation-form/task-creation-form.component.ts`

**Responsibilities**:
- Render task creation form (inline and modal variants)
- Implement real-time validation with Angular Reactive Forms
- Handle optimistic UI updates
- Subscribe to WebSocket task creation events
- Display validation errors and success feedback

**Key Features**:
- FormGroup with validators: title (required, maxLength 200), description (maxLength 2000), assignee (optional), priority (required enum), dueDate (optional, future date), tags (array)
- Async validators for duplicate title checking
- Keyboard shortcuts (Ctrl+Enter to submit)
- Entrance animations for remote tasks (300ms fade-in)

**ADR Mapping**: DEMO-1887

#### 4.1.2 TaskAnimationService
**Responsibilities**:
- Manage task appearance animations
- Handle stagger delays for multiple simultaneous tasks
- Provide smooth transitions for real-time updates

### 4.2 Backend Services

#### 4.2.1 TaskValidationService
**Location**: `src/main/java/com/scib/service/TaskValidationService.java`

**Responsibilities**:
- Comprehensive task validation using Bean Validation annotations
- Custom validators for business rules
- Async validation for duplicate title checking
- Return detailed ValidationResult objects

**Validation Rules**:
- Title: not empty, unique within board, max 200 characters
- Description: max 2000 characters
- Assignee: must exist and have board access
- Priority: valid enum value
- DueDate: future date if provided
- Column: must exist in board

**ADR Mapping**: DEMO-1888

#### 4.2.2 TaskCreationService
**Location**: `src/main/java/com/scib/service/TaskCreationService.java`

**Responsibilities**:
- Orchestrate task creation workflow
- Integrate with TaskValidationService
- Persist tasks to database with transaction management
- Publish events to Redis pub/sub
- Handle bulk task creation with batch processing

**Key Methods**:
- `createTask(TaskCreationRequest request)`: Single task creation
- `createTasksBatch(List<TaskCreationRequest> requests)`: Bulk creation with batch processing

**ADR Mapping**: DEMO-1889, DEMO-1891

#### 4.2.3 WebSocketBroadcastService
**Responsibilities**:
- Subscribe to Redis pub/sub channels
- Broadcast task creation events to connected clients
- Manage WebSocket connections and sessions
- Handle connection failures and reconnection

## 5. Data Design

### 5.1 Database Schema Updates

#### 5.1.1 Tasks Table Constraints
**Migration**: `V1.53__task_creation_constraints.sql`

**New Constraints**:
```sql
-- Unique constraint to prevent duplicate titles per board
ALTER TABLE tasks ADD CONSTRAINT unique_task_title_per_board 
    UNIQUE (board_id, title);

-- Check constraints for data quality
ALTER TABLE tasks ADD CONSTRAINT check_title_not_empty 
    CHECK (length(trim(title)) > 0);

ALTER TABLE tasks ADD CONSTRAINT check_description_length 
    CHECK (length(description) <= 2000);

-- Foreign key with cascading behavior
ALTER TABLE tasks ADD CONSTRAINT fk_assignee_user 
    FOREIGN KEY (assignee_id) REFERENCES users(id) ON DELETE SET NULL;
```

**New Indexes**:
```sql
-- Performance indexes for common queries
CREATE INDEX idx_tasks_created_at ON tasks(created_at);
CREATE INDEX idx_tasks_board_created ON tasks(board_id, created_at DESC);
```

## 6. Interface Design

### 6.1 REST API Endpoints

#### 6.1.1 Single Task Creation
**Endpoint**: `POST /api/boards/{boardId}/tasks`

**Request Schema**:
```json
{
  "title": "string (required, max 200)",
  "description": "string (optional, max 2000)",
  "assigneeId": "uuid (optional)",
  "priority": "enum [LOW, MEDIUM, HIGH, CRITICAL]",
  "dueDate": "ISO8601 datetime (optional)",
  "tags": ["string"],
  "columnId": "string (required)"
}
```

**Response Schema**:
```json
{
  "id": "uuid",
  "title": "string",
  "description": "string",
  "boardId": "uuid",
  "assigneeId": "uuid",
  "assigneeName": "string",
  "priority": "enum",
  "dueDate": "ISO8601 datetime",
  "tags": ["string"],
  "columnId": "string",
  "position": "integer",
  "createdAt": "ISO8601 datetime",
  "createdBy": "uuid",
  "createdByName": "string"
}
```

#### 6.1.2 Bulk Task Creation
**Endpoint**: `POST /api/boards/{boardId}/tasks/batch`

**Request Schema**:
```json
{
  "tasks": [
    {
      "title": "string",
      "description": "string",
      "assigneeId": "uuid",
      "priority": "enum",
      "dueDate": "ISO8601 datetime",
      "tags": ["string"],
      "columnId": "string"
    }
  ]
}
```

### 6.2 WebSocket Message Format

#### 6.2.1 Task Creation Event
```json
{
  "type": "TASK_CREATED",
  "boardId": "uuid",
  "task": {
    "id": "uuid",
    "title": "string",
    "description": "string",
    "assigneeId": "uuid",
    "assigneeName": "string",
    "priority": "enum",
    "dueDate": "ISO8601 datetime",
    "tags": ["string"],
    "columnId": "string",
    "position": "integer",
    "createdAt": "ISO8601 datetime",
    "createdBy": "uuid",
    "createdByName": "string"
  },
  "timestamp": "ISO8601 datetime"
}
```

#### 6.2.2 Batch Creation Event
```json
{
  "type": "TASKS_BATCH_CREATED",
  "boardId": "uuid",
  "tasks": ["array of task objects"],
  "batchId": "uuid",
  "timestamp": "ISO8601 datetime"
}
```

## 7. Non-Functional Aspects

### 7.1 Performance Requirements
- **Response Time**: Task creation API response < 200ms for 95% of requests
- **Synchronization Latency**: Tasks appear on remote clients within 500ms
- **WebSocket Broadcast**: Event publishing to Redis < 50ms
- **Bulk Processing**: Support > 100 tasks/second for batch operations
- **Database Performance**: Optimized queries with proper indexing

### 7.2 Scalability
- **Horizontal Scaling**: Stateless services support multiple instances
- **Load Balancing**: WebSocket connections distributed across pods
- **Database Scaling**: Read replicas for query performance
- **Redis Clustering**: Pub/sub scaling for high-volume events
- **Batch Size Limits**: Maximum 1000 tasks per batch request

### 7.3 Reliability
- **Transaction Management**: ACID compliance for task creation
- **Event Delivery**: At-least-once delivery guarantee for WebSocket events
- **Failure Recovery**: Automatic retry mechanisms for failed operations
- **Data Consistency**: Optimistic locking for concurrent updates
- **Circuit Breakers**: Fault tolerance for external service calls

### 7.4 Security
- **Authentication**: JWT-based user authentication
- **Authorization**: Role-based access control (RBAC) for board access
- **Input Validation**: Comprehensive server-side validation
- **SQL Injection Prevention**: Parameterized queries and ORM usage
- **XSS Protection**: Output encoding and CSP headers
- **Data Encryption**: TLS 1.3 for data in transit

### 7.5 Compliance
- **GDPR**: User data protection and privacy controls
- **Audit Logging**: All task creation events logged with user context
- **Data Retention**: Configurable retention policies
- **Access Logging**: Comprehensive audit trail for compliance

## 13. Technical Risks

### 13.1 High-Priority Risks

| Risk | Impact | Likelihood | Mitigation Strategy |
|------|--------|------------|--------------------|
| WebSocket connection instability | High | Medium | Implement connection retry logic, fallback to polling |
| Database performance degradation | High | Low | Implement proper indexing, query optimization, connection pooling |
| Redis pub/sub message loss | Medium | Low | Implement message acknowledgment, persistent queues for critical events |
| Bulk operation memory issues | Medium | Medium | Implement streaming processing, batch size limits |

### 13.2 Medium-Priority Risks

| Risk | Impact | Likelihood | Mitigation Strategy |
|------|--------|------------|--------------------|
| Frontend animation performance | Medium | Medium | Optimize animations, implement performance monitoring |
| Validation service bottleneck | Medium | Low | Implement caching, async validation where possible |
| Cross-browser WebSocket compatibility | Low | Medium | Implement WebSocket polyfills, progressive enhancement |
